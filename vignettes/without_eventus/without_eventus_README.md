# without_eventus

These scripts implement the same cleaning and analysis pipelines as
the eventus vignette series using standard pandas — no eventus. They
were generated by a large language model given identical specifications
and refactored to use global constants. This is the fairest comparison
against eventus — not a first draft, but competent, well-structured
script code.

They load from the same synthetic datasets so readers can run both
side by side and compare output.

## What these scripts demonstrate

All implemented scripts are functional and produce correct output.
The comparison is not about correctness — it is about what the
script-based paradigm structurally cannot provide regardless of
code quality:

- **No versioned config.** Cleaning and aggregation parameters live
  as global constants and hardcoded lambdas. A collaborator reading
  the output cannot reconstruct the decisions without reading the
  source code.

- **Hardcoded column references throughout.** Rename a column and
  you must find every reference manually.

- **Adaptation requires rewriting.** Adding a grouping column, a
  new descriptor, or changing an aggregation rule requires editing
  the core logic — not a config change.

- **Hospital-aware deduplication works by coincidence.** The ED
  visits script uses whole-row dedup, which happens to preserve
  same-date different-hospital records. This is not declared intent.
  `also_defined_by` makes the intent explicit.

- **Type validation is absent.** Nothing prevents applying `mean`
  to `mobility_status`. eventus validates at construction.

- **Silent warnings instead of specific errors.** The Chapter 4
  script produced a pandas `UserWarning` that continues execution
  silently. eventus raises with a specific, actionable message
  before any data is touched.

- **Some things cannot be reproduced faithfully.** The Chapter 5
  matplotlib script produces a plot — but with a normalized x-axis,
  not real dates. The escalating complexity across chapters — ~150
  lines in Chapter 1, ~275 lines in Chapter 4, 165 lines with a
  fundamental limitation in Chapter 5 — is itself the argument.

These are properties of the paradigm, not of the code quality.
